02. Project Overview

Why This Engine Exists

This blackjack engine solves a common gap in game demos: most portfolio blackjack projects either stop at simple console play or bury the rules inside UI code. This project instead separates the core game engine, statistical analytics, strategy recommendation logic, and JavaFX presentation into distinct layers.

The result is not just a playable casino-style desktop app. It is an explainable algorithmic system that can answer what move is recommended, why it is recommended, and how risky the current state is while preserving a reusable engine that can later power tests, bots, Monte Carlo analysis, or a different UI front-end.

Why Strategy Analysis Matters

  • Best Moveconverts basic-strategy rules into explainable decisions for the current hand.
  • Expected EVprovides a normalized heuristic estimate of long-term hand quality.
  • Bust Riskquantifies immediate risk when drawing one additional card.
  • Dealer Bust Chanceexposes the pressure created by a weak dealer up-card.
  • Recommendation Analyticsmeasures whether following advice correlates with better outcomes across the session.

Engineering Assumptions

  • A fresh 52-card deck is rebuilt and shuffled at the beginning of every round.
  • The dealer stands on all 17 values, including soft 17, because the draw rule is value < 17.
  • Natural blackjack is detected immediately after the opening deal.
  • Natural blackjack currently pays even money in the engine: payout = 2 × bet, not 3:2.
  • The current implementation allows one split per round, which simplifies multi-hand sequencing and UI management.

03. Core OOP Architecture

Class Responsibilities And Layering

Class Primary Responsibility Key Design Notes
Card Immutable suit/rank object with nominal blackjack value and ace detection. Separates symbolic identity from hand logic. Ace flexibility is handled in Hand, not here.
Deck Builds a standard 52-card deck, shuffles it, and deals in sequence. Reset/shuffle is round-scoped. The current engine uses a fresh deck every round rather than a reusable shoe.
Hand Tracks cards, computes best blackjack value, soft state, blackjack, bust, and split eligibility. The ace downgrade loop is the core of correct value evaluation.
Player Owns bankroll and enforces valid bet commitment / payout return. Bankroll mutation is centralized here, which prevents duplicate UI-side chip changes.
Dealer Owns dealer hand and dealer draw rule. Dealer policy is deliberately tiny: draw while value < 17.
GameEngine Coordinates the full round lifecycle: bet commit, initial deal, action branching, dealer turn, settlement, statistics. This is the main orchestration boundary. UI code reads from it but does not own rule transitions.
RecommendationAnalytics Tracks whether users follow or ignore recommendations and what outcomes those choices lead to. Decisions are recorded per hand and attached to final HandResult objects after Settlement.
StrategyAdvisor Produces best move, reason, confidence, EV, bust risk, and dealer bust chance. It is rule-based and explainable, with fast heuristics rather than Monte Carlo search.
RoundResult Captures the resolved state of a round: outcome, chip delta, total bet, total payout, and per-hand details. Acts as the handoff object to UI overlays, statistics, and analytics.
SessionStatistics Implemented as GameStatistics plus snapshot DTOs for persistence and export. Tracks wins, losses, pushes, blackjacks, split usage, double-down usage, win streaks, and total P/L.

04. Game Flow Algorithm

Round Lifecycle And Branching

Step-by-Step Lifecycle

  1. Bankroll + bet validation. The player must have enough chips to commit the opening stake.
  2. Fresh deck initialization. The engine creates a new deck and shuffles it every round.
  3. Opening hand creation. The engine builds one PlayerHand and deals two player cards and two dealer cards.
  4. Natural blackjack check. Before normal play begins, the engine resolves immediate blackjack outcomes.
  5. Player decision loop. The active hand can hit, stand, double down, or split, subject to availability.
  6. Multi-hand sequencing. Split hands are queued sequentially, and the active index advances to the next open hand.
  7. Dealer turn. If at least one player hand survives, the dealer draws until the policy threshold is met.
  8. Settlement. Each hand becomes a HandResult; the round becomes a RoundResult.
  9. Session update. Statistics, recommendation analytics, overlays, chips, and labels refresh from the resolved result object.
Initial dealing split between engine and UI. The engine mutates authoritative game state immediately when a round begins. The UI then plays the opening sequence against that already-resolved state, which keeps animations deterministic without moving rule logic into JavaFX.

Payout Settlement Contract

B = committed bet for a hand
P = payout returned to bankroll
Δchips = P - B

Normal win:        P = 2B,  Δchips = +B
Push:              P = B,   Δchips = 0
Loss / bust:       P = 0,   Δchips = -B
Natural blackjack: current implementation also uses P = 2B

05. Strategy Recommendation Algorithm

Decision Engine, Heuristics, And Analysis Fields

Recommendation Pipeline

  1. Read the currently active hand and the dealer's visible up-card.
  2. Map the dealer card into a strategy value where face cards become 10 and ace becomes 11.
  3. If split is available and the hand is a pair, evaluate split rules first.
  4. If not split, branch into soft total or hard total strategy tables.
  5. Return the action plus reason string.
  6. Derive analysis metadata: confidence, win tendency, risk level, EV, bust risk, dealer bust chance.
Why split rules come first. Pair splitting is not merely a total-based decision. It restructures the game tree into two separate betting branches, so it must be evaluated before soft or hard total logic.

Confidence

Confidence is highest for canonical strategy states such as splitting aces or eights, doubling hard 11 against a non-ace, or hitting hard totals that are clearly below 12. Edge cases degrade to medium confidence.

Risk & Tendencies

Risk rises for double down, split branches, and hard hit states near bust thresholds. Win tendency is based on hand softness, total strength, and dealer weakness or strength.

06. Mathematical Logic

Formulas, Heuristics, And Normalized Metrics

Ace-Aware Hand Value

total = Σ nominalCardValue
aces = count(Ace)

while total > 21 and aces > 0:
    total = total - 10
    aces = aces - 1

bestValue = total
softHand = at least one Ace remains valued as 11

Bust Risk

BustRisk(hand) =
    Σ [ w(r) × I( bust(hand + r) ) ] / Σ w(r)

where:
    w(2..9) = 4
    w(10/J/Q/K bucket) = 16
    w(A) = 4

The strategy engine simulates a one-card draw against a weighted abstract rank distribution, which is fast enough for every UI refresh.

Dealer Bust Chance

DealerBustChance(upCard) = lookup(upCard)

2 → 0.35   3 → 0.37   4 → 0.40   5 → 0.42   6 → 0.42
7 → 0.26   8 → 0.24   9 → 0.23   10 → 0.21   A → 0.17

Expected Value Heuristic

EV =
    softBias
  + dealerWeaknessBias
  + totalStrengthBias
  + actionBonus
  - actionBustPenalty
  + 0.42 × DealerBustChance

Clamp(EV, -0.65, +0.65)

This is a normalized engineering heuristic, not exact combinatorial EV. It is intentionally fast, explainable, and stable under frequent UI refresh.

Win Rate, Session Profit, Recommendation Accuracy

ResolvedHands = wins + losses + pushes
WinRate = wins / ResolvedHands

SessionProfit = Σ roundChipDelta

FollowRate = totalRecommendationsFollowed / totalRecommendationsShown

RecommendationAccuracy =
    (followedWins + followedPushes) / resolvedFollowedCount

Payout And Bankroll Integrity

On round start:
    player.commitBet(B)

On split / double:
    player.commitBet(additionalStake)

At settlement:
    player.receivePayout(totalPayoutAmount)

ChipDelta = totalPayoutAmount - totalCommittedBet

The engine performs payout exactly once during finishRound(...), which prevents duplicate bankroll updates from UI actions.

Metric Interpretation Engineering Tradeoff
Expected EV Normalized directional estimate of long-term value for the recommended action. Fast enough for live UI updates, but intentionally approximate.
Bust Risk Probability that drawing one more card immediately busts the current hand. Uses weighted ranks, not exact remaining-deck composition.
Dealer Bust Chance Precomputed weakness estimate derived from the visible dealer up-card. Near constant-time and stable; ideal for UI explanation.
Confidence How strongly the decision matches canonical strategy patterns. Categorical, human-readable, intentionally not statistical confidence.

07. UI-State Synchronization

Keeping JavaFX Presentation Aligned With Engine State

Synchronization Strategy

  • Engine-first state. Rules update first; animations render the already-valid state.
  • Single refresh hub. updateUi() coordinates headers, cards, recommendation panel, footer metrics, controls, and round-end effects.
  • Animation gating. The controller uses explicit flags such as animationInProgress and initialDealAnimationInProgress to keep buttons disabled during transitions.
  • Split-hand awareness. The active hand index drives highlighting, score display, and recommendation context.
  • Resolved-state handoff. Once a round is over, recommendation values switch from live analysis to a safe “round complete” state, while overlays and settlement chip animations take over.

08. Performance & Optimization

Optimization Opportunities And Rendering Efficiency

Current Performance Characteristics
  • Fresh deck + shuffle per round keeps state simple and deterministic.
  • PNG card images are cached in memory after first load.
  • Dealer bust probabilities are constant-time lookups.
  • Bust risk runs against a ten-rank weighted distribution, not full deck permutations.
Optimization Opportunities
  • (hand signature, dealer up-card, action) by (hand signature, dealer up-card, action).
  • Upgrade from per-round deck rebuilds to a multi-deck shoe for casino realism.
  • Add split recursion caching when multiple re-splits are introduced.
  • Batch some JavaFX node refreshes when large split layouts are active.
Rendering Optimizations
  • Use a dedicated animation layer so transient cards and chips do not relayout the entire scene.
  • Preserve consistent card dimensions to avoid layout thrash during dealing.
  • Keep card images rasterized and cached to avoid runtime SVG decoding cost.
  • Separate stable felt/background layers from dynamic overlays.

10. Final Engineering Summary

This project demonstrates a strong full-stack engineering profile because it treats a seemingly simple game as a serious systems problem. The most portfolio-worthy characteristic is not that the application can deal cards. It is that the application can justify decisions, measure behavioral outcomes, animate state without corrupting logic, and persist a meaningful session model. Those qualities make the project suitable for GitHub showcase material, system design discussion, and interview walkthroughs that emphasize architecture, correctness, and product thinking.
